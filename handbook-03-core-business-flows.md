# 第 3 册：核心业务流

## 这册要解决什么问题

前两册讲的是“平台分层”和“核心对象”。这册开始看“业务是怎么流动起来的”。

你需要理解的不是页面长什么样，而是这些典型流程：

- 控制流
- 联动流
- 告警流
- 报修流
- 运营流

学完后你应该能做到：

- 用流程视角解释智能家居平台在做什么
- 识别哪些流属于底层，哪些流属于上层
- 为 PoC 挑选合适的演示路径

## 一、控制流

最基础的业务流是控制流。

一个典型控制流是：

1. 用户发起控制请求
2. 平台校验权限
3. 平台下发设备指令
4. 设备执行
5. 设备返回状态或平台同步结果
6. 页面展示控制结果

控制流关注的是：`把动作执行出去，并确认结果`。

这条流要求高可靠、可追溯，通常必须放在底层平台能力里。

## 二、联动流

联动流是“某个条件满足后，自动触发一组动作”。

典型例子：

1. 烟感设备上报异常事件
2. 规则引擎识别条件满足
3. 自动触发风机、广播、门禁等联动
4. 记录联动结果
5. 通知相关人员

联动流关心的是：`实时自动执行一组规则动作`。

这类流通常不能依赖桌面操作或人工确认，否则会失去时效性。

## 三、告警流

告警流是“平台识别异常并组织处理”的过程。

一个常见告警流：

1. 设备上报异常事件
2. 平台判断是否形成告警
3. 告警入库并展示
4. 通知值班人
5. 人工确认或自动派单
6. 跟踪处理结果
7. 告警关闭

这里要注意，前半段更偏底层能力，后半段更偏运营能力。

## 四、报修流

报修流通常跨越设备平台和业务管理平台。

例如：

1. 用户反馈问题
2. 客服或系统初步判断设备类型与位置
3. 查询设备当前状态与历史异常
4. 生成工单
5. 指派处理人
6. 处理完成后回填结果

这类流程特别适合引入 Agent 作为辅助入口，因为它涉及：

- 信息查询
- 历史汇总
- 规则判断
- 工单流转

## 五、运营流

运营流不是控制设备，而是“围绕平台使用产生的管理动作”。

例如：

- 每日异常汇总
- 每周设备在线率报表
- 项目维度对比分析
- 批量通知相关人员
- 导出运营数据

这类流通常最适合做自动化，因为它们：

- 流程相对稳定
- 需要跨系统
- 对实时性要求没那么极端
- 经常有人工重复操作

## 六、为什么你要学业务流

因为将来和朋友沟通时，不能只说“这个平台能不能做”，而要说：

- 它想优化的是哪一条流
- 这条流卡在什么地方
- 该由哪一层解决
- 是否适合拿来做 PoC

## 七、PoC 最适合挑哪几条流

如果你想快速验证“实在Agent 有没有价值”，建议优先挑：

1. 告警流中的信息汇总与通知
2. 报修流中的状态查询与工单生成
3. 运营流中的日报、周报、月报

不建议一开始拿“实时联动控制”做 PoC，因为那不是它最稳的战场。

## 八、实在Agent 最适合插在哪

通常是在这些环节：

- 查询与汇总
- 多系统串联
- 文本总结
- 报表生成
- 工单触发

不建议插在这些环节：

- 核心指令下发主链路
- 实时联动主闭环
- 安全级别高的无确认控制动作

## 九、自测题

2. 为什么联动流不适合依赖 RPA 主导？
3. 哪 3 条业务流最适合用来验证 Agent 的价值？

## 十、练习任务

自己写出下面 3 条流程的步骤：

1. 用户远程开灯
2. 烟感异常后的自动处理
3. 每日离线设备报告生成

然后标注每一步属于哪一层。
