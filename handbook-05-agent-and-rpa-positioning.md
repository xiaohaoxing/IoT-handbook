# 第 5 册：Agent / RPA 的定位与边界

## 这册要解决什么问题

现在你已经知道平台怎么分层，也知道 IoT 和运营平台的区别。这册专门回答一个核心问题：

`Agent / RPA 到底应该放在哪里？`

学完后你应该能做到：

- 说清楚 Agent 和 RPA 分别擅长什么
- 判断它们在智能家居方案中适不适合用
- 知道什么时候该用，什么时候不该用

## 一、先把 Agent 和 RPA 分开理解

### Agent

Agent 更偏“理解任务、组织步骤、调用能力”。

它擅长：

- 理解自然语言
- 做信息归纳
- 基于规则或上下文做简单判断
- 调用接口、流程、知识库、工具

### RPA

RPA 更偏“模仿人在软件里的操作”。

它擅长：

- 点击网页
- 填表
- 复制粘贴
- 跨多个旧系统搬运数据
- 没有 API 时补自动化

### 两者结合

把两者结合后，常见模式就是：

- Agent 负责理解和编排
- RPA 负责执行某些界面动作

## 二、为什么它们适合做上层能力

因为上层运营和业务流程里，经常会出现这些问题：

- 人工重复查询
- 多系统来回切换
- 没有统一接口
- 需要文字总结
- 需要自然语言入口

这正好是 Agent / RPA 的强项。

## 三、为什么它们不适合做底层控制核心

因为底层控制核心需要：

- 低延迟
- 高可用
- 强一致性
- 标准化协议
- 明确审计链路

而 Agent / RPA 的弱点通常在于：

- 执行链路更长
- 依赖上下文和外部界面
- 对页面变化敏感
- 很难成为严格实时闭环

## 四、在智能家居里最适合的用法

### 适合

- 查状态
- 汇总异常
- 生成日报周报
- 自动通知
- 自动建工单
- 串联多个后台系统
- 补齐缺少 API 的外围系统

### 不适合

- 主导设备实时控制
- 主导高风险安全动作
- 做设备接入协议层
- 做核心状态同步系统
- 作为唯一联动引擎

## 五、你要特别注意“控制类操作”

控制类操作不是不能接入 Agent，而是必须分级。

建议原则：

- 查询类：可以自然语言开放
- 低风险控制类：建议二次确认
- 高风险控制类：必须显式审批或仅走底层系统

例如：

- “查一下 3 楼空调状态”可以直接做
- “把 3 楼所有空调调到 24 度”建议确认一次
- “远程开门”这类动作就要更严

## 六、如何向别人解释它的价值

你可以这样说：

“它不是替代设备平台，而是给已有平台加一个更自然的入口和自动化层，让查询、汇总、工单、报表、跨系统协作更省人。”

这句话基本能把预期拉正。

## 七、适用判断表

当你评估一个场景时，先问：

1. 这是查信息还是控设备？
2. 这是分钟级流程，还是秒级实时动作？
3. 有没有现成 API？
4. 参与系统多不多？
5. 需要给非技术人员用吗？

如果答案偏向：

- 查询
- 分钟级
- 系统多
- 需要自然语言和自动化

那 Agent / RPA 通常就比较合适。

## 九、练习任务

把下面场景判断成“适合 / 不适合 / 需条件限制”：

- 自动汇总昨日离线设备
- 自然语言查询某房间状态
- 远程开门
- 烟感触发后主导联动
- 没有 API 的老旧物业系统数据录入
