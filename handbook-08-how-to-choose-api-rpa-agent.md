# 第 8 册：API、RPA、Agent 的选型方法

## 这册要解决什么问题

到这里你已经具备一个很关键的前提：知道底层平台和上层平台的边界，也知道基本集成手段。

现在要解决的是：`一个具体需求来了，到底该用 API、RPA，还是 Agent？`

## 一、先记最重要的原则

`能用 API 就先用 API；需要自然语言入口或智能编排时再加 Agent；没有标准接口时再考虑 RPA。`

这是默认顺序。

## 二、三者分别擅长什么

### API

擅长：

- 标准化调用
- 稳定自动化
- 系统对系统集成
- 结构化数据处理

### RPA

擅长：

- 没有接口时补操作
- 老系统页面自动化
- 人工录入类流程

### Agent

擅长：

- 理解自然语言
- 做任务拆解
- 多能力编排
- 生成摘要、报告、解释

## 三、一个需求该怎么判断

你可以按这 5 个问题判断：

1. 有没有现成 API？
2. 这个动作是查询，还是控制？
3. 需要自然语言交互吗？
4. 是否涉及多个系统？
5. 风险高不高？

## 四、几个典型判断例子

### 场景 1：查询昨日离线设备

建议：

- 底层用 API 查数据
- 上层可以加 Agent 做自然语言入口和摘要

### 场景 2：把老旧物业系统中的信息录入新系统

建议：

- 优先看有无接口
- 没有接口时，RPA 很可能是合理方案

### 场景 3：自然语言问“今天哪个项目异常最多”

建议：

- 用 Agent 做入口
- 背后调用 API 拉数据

### 场景 4：远程开门

建议：

- 主能力必须来自底层平台 API
- 是否允许 Agent 触发，要看审批和确认设计

## 五、常见误区

### 误区 1：Agent 能做一切

不对。Agent 不是替代所有系统，而是调用或编排已有能力。

### 误区 2：RPA 是万能补丁

不对。RPA 是重要补丁，但长期核心流程如果全靠页面点点点，维护成本会很高。

### 误区 3：有自然语言入口就等于智能化

不对。真正决定成败的还是底层能力是否稳定、接口是否可用、流程是否清楚。

## 六、一个简单选型表

### 优先 API

- 有标准接口
- 结构化数据明确
- 需要稳定高频执行

### 考虑 Agent

- 需要自然语言查询
- 需要总结解释
- 需要跨系统编排

### 考虑 RPA

- 没有接口
- 只能通过页面操作
- 流程规则相对稳定
