# 第 7 册：API / Webhook / JSON 基础

## 这册要解决什么问题

前面你建立了产品和方案视角，这册开始补最必要的集成基础。

你不用一下子学成开发，但至少要看懂下面这些词在方案里是什么意思：

- API
- Webhook
- JSON
- Token / 鉴权

## 一、什么是 API

API 可以先简单理解成：`系统给外部开放的一组标准能力入口`。

例如，一个设备平台可以提供：

- 查询设备列表
- 查询设备状态
- 下发控制指令
- 查询告警记录

这些都可以通过 API 提供给外部系统调用。

## 二、为什么 API 很重要

因为它是最稳定、最标准的系统集成方式。

如果一个动作有现成 API，一般优先走 API，而不是 RPA 点页面。

原因很简单：

- 更稳
- 更快
- 更容易追踪
- 更适合自动化

## 三、什么是 Webhook

Webhook 可以理解成：`当系统里发生某件事时，主动把消息推给另一个系统`。

比如：

- 告警发生时推送给工单系统
- 工单关闭时推送给通知系统

API 更像“我去问你要数据”。  
Webhook 更像“我有事主动告诉你”。

## 四、什么是 JSON

JSON 是系统之间交换数据时最常见的一种结构化格式。

你可以把它看成一种有层级的键值结构，比如：

```json
{
  "deviceId": "AC-301",
  "status": "online",
  "targetTemperature": 24
}
```

你不用先学编程，但要能看懂：

- 字段名是什么
- 当前值是什么
- 数据层级是什么

## 五、什么是鉴权

系统不会无条件开放 API，通常会做鉴权。

最常见的是：

- Token
- API Key
- 用户身份登录态

你作为方案侧至少要意识到：

- 能不能调用，不只是“有没有接口”
- 还要看“有没有权限”

## 六、为什么这对 Agent/RPA 很关键

你后面判断集成方式时，核心问题之一就是：

`这个动作能不能直接调 API？`

如果能：

- Agent 可以直接调
- 流程更稳定
- 审计更清晰

如果不能：

- 才考虑用 RPA 补界面自动化

## 七、你现在至少要能看懂什么

以后别人给你一份接口文档，你至少要能看懂：

- 这个接口是干什么的
- 输入参数有哪些
- 返回结果有哪些
- 是否涉及控制动作
- 是否需要权限

## 八、一个简单例子

查询设备状态的接口，大致会包含：

- 请求路径
- 请求方式
- 身份认证
- 请求参数
- 返回字段

你现在不用记技术细节，但要建立一个意识：

`方案能不能落地，往往取决于底层接口是否可用。`

## 十、练习任务

假设有一个“查询设备状态”的接口，请你自己用文字描述它可能包含：

- 输入什么
- 返回什么
- 会被谁调用
- 适合用在什么场景
