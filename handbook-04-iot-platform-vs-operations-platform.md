# 第 4 册：IoT 平台与运营平台的区别

## 这册要解决什么问题

很多方案讨论会混淆两个东西：

- IoT 平台
- 运营平台

这册的目标就是把这两个概念拆开。

学完后你应该能做到：

- 解释它们服务的对象不同
- 解释它们关注的问题不同
- 判断一个需求更像平台底座能力，还是业务管理能力

## 一、IoT 平台在解决什么问题

IoT 平台的核心任务是：`让设备可靠接入、可被理解、可被控制、可被追踪`。

它主要面向的是：

- 设备
- 网关
- 协议
- 状态
- 指令
- 规则

典型关键词：

- 接入
- 建模
- 控制
- 同步
- 规则引擎
- 时序状态

## 二、运营平台在解决什么问题

运营平台的核心任务是：`让人能高效使用设备能力完成业务目标`。

它主要面向的是：

- 运维人员
- 客服人员
- 物业/运营人员
- 管理层

典型关键词：

- 看板
- 工单
- 报表
- 协同
- 通知
- 统计分析

## 三、两者的差别不是“技术高低”，而是职责不同

IoT 平台更像“机器层和能力层”。  
运营平台更像“人和业务层”。

你可以这样记：

- IoT 平台回答：设备现在怎样，能不能控制，规则能不能跑
- 运营平台回答：谁来处理异常，如何追踪问题，怎样提升效率

## 四、为什么这点很重要

因为实在Agent 这样的产品，一般更自然地贴近运营平台，而不是 IoT 核心底座。

它更擅长：

- 帮人查
- 帮人汇总
- 帮人串流程
- 帮人自动处理重复动作

而不是：

- 接设备
- 跑核心协议
- 维护时序状态
- 主导实时联动

## 五、一个具体例子

假设你问：

“我想知道今天哪些门锁异常最多，并自动生成处理任务。”

这件事拆开看：

- 门锁状态采集、异常识别：更偏 IoT 平台
- 异常汇总、生成任务、通知人员：更偏运营平台

这时 Agent 可以很自然地放在后半段。

## 六、和朋友沟通时怎么避免跑偏

建议你多用这些追问：

1. 这是设备能力问题，还是业务运营问题？
2. 这个动作要求实时闭环，还是允许分钟级处理？
3. 这是给设备用的，还是给人用的？
4. 这是高可靠控制链路，还是上层辅助能力？

只要这 4 个问题问明白，方案大方向一般不会偏。

## 七、判断口诀

你可以先记一句简化判断：

`设备能不能接、控、报、联动，是 IoT 平台的事；人怎么查、看、处理、协同，是运营平台的事。`

## 八、自测题

1. IoT 平台和运营平台各自主要服务谁？
2. 为什么“异常工单流转”更像运营平台能力？
3. 为什么“实时设备联动”不能作为 Agent 层主逻辑？

## 九、练习任务

把下面这些需求分类：

- 批量看离线设备
- 实时门磁联动灯光
- 查询某项目本周设备在线率
- 设备协议接入
- 自动创建异常工单
- 门锁开门指令下发

分别归到：

- IoT 平台
- 运营平台
- 两者协同
