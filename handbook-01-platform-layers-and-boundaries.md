# 第 1 册：智能家居平台分层与边界

## 这册要解决什么问题

很多人一开始接触智能家居、IoT、Agent、RPA 时，最容易犯的错误不是不会操作，而是“把平台边界想错了”。

比如：

- 把 AI Agent 当成设备控制平台
- 把设备接入平台当成普通后台
- 把报表、工单、运营能力和设备指令控制混在一起

这册的目标，就是先把这些边界拆开。

学完后你应该能做到：

- 说出智能家居管理平台的主要分层
- 解释每一层的职责
- 判断某个需求应该放在哪一层实现

## 一、先建立一个最基本的整体视图

可以把一个智能家居管理平台先粗略看成 5 层：

1. 设备接入层
2. 设备能力层
3. 业务编排层
4. 运营管理层
5. AI / Agent / 自动化层

这 5 层不是所有项目都会严格拆成 5 个独立系统，但职责通常都存在。

### 1）设备接入层

这一层关心的是“设备怎么连进来”。

典型职责：

- 设备注册
- 鉴权
- 协议接入
- 网关接入
- 上下行通信
- 在线/离线判断

你可以把它理解成：这是平台和物理设备打交道的入口层。

如果这一层不稳定，后面所有上层能力都会漂。

### 2）设备能力层

这一层关心的是“平台如何理解设备，以及如何控制设备”。

典型职责：

- 设备档案
- 设备模型
- 属性读取
- 指令下发
- 事件上报
- 状态同步

这一层通常会定义一个核心对象体系：

- 设备是什么
- 它有哪些属性
- 它能接受哪些指令
- 它会上报哪些事件

这层是整个智能家居平台的“事实基础”。

### 3）业务编排层

这一层关心的是“如何把设备能力组织成业务逻辑”。

典型职责：

- 场景联动
- 自动规则
- 定时任务
- 异常处理流程
- 多设备协同

例如：

- 人离开房间后自动关灯
- 温度高于阈值时打开空调
- 烟感告警时触发一组联动动作

这里已经不是单个设备的控制，而是“规则和流程”的组合。

### 4）运营管理层

这一层关心的是“人怎样使用、管理和运营这个平台”。

典型职责：

- 后台管理
- 项目/楼栋/房间管理
- 用户与权限
- 告警看板
- 工单系统
- 报表分析
- 客服与运维协作

很多管理平台真正被频繁使用的，往往就是这一层。

### 5）AI / Agent / 自动化层

这一层关心的是“如何让人更高效地使用前面几层能力”。

典型职责：

- 自然语言查询
- 智能总结
- 自动生成报表
- 跨系统流程编排
- 对老旧系统做自动化操作
- 把复杂操作封装成可复用任务

这就是实在Agent 这类产品更适合待的位置。

它不是设备平台的地基，更像是设备平台之上的智能操作层。

## 二、为什么必须分层

因为不同问题的工程约束完全不同。

### 底层问题关心什么

设备与控制相关的底层系统，通常关心：

- 稳定性
- 实时性
- 状态一致性
- 高并发
- 设备协议兼容
- 指令成功率

例如，开门、关灯、告警联动、空调控制，这些都不能靠“模仿人点按钮”的方式来做核心闭环。

### 上层问题关心什么

运营与自动化相关的上层系统，更关心：

- 易用性
- 跨系统协同
- 查询与统计
- 流程自动化
- 对非技术人员友好
- 低成本试错

例如，日报、工单流转、异常通知、信息汇总、智能问答，这些就很适合放在 Agent/RPA 层。

## 三、怎样判断一个能力放在哪一层

你可以用一个很简单的方法判断：先问“这是设备核心能力，还是业务使用能力”。

### 应该放在底层 IoT 平台的能力

- 设备接入
- 设备建模
- 状态同步
- 指令控制
- 联动规则引擎
- 设备告警核心逻辑
- 权限与审计底座

这些能力的共同点是：它们必须稳定、可控、可追溯，而且不能依赖桌面自动化。

### 适合放在上层 Agent/RPA 平台的能力

- 自然语言查状态
- 告警信息汇总
- 报表自动生成
- 工单自动流转
- 对多个后台系统做串联
- 对没有 API 的外围系统做补充自动化

这些能力的共同点是：它们更像“使用平台”和“运营平台”的增强层。

## 四、把米家接入作为例子来理解

假设你们已经接入了大量米家设备，而且基础操作都正常。

那说明下面这些事情已经基本有着落：

- 设备已进平台
- 基础控制可用
- 状态能读
- 一些标准场景可以执行

这时再引入实在Agent，合理的位置通常是：

- 作为统一查询入口
- 作为运营自动化入口
- 作为跨系统串联工具
- 作为面向客服/运营/实施人员的智能助手

不合理的位置通常是：

- 替代米家设备控制底座
- 作为实时联动主引擎
- 作为主状态存储系统

## 五、和朋友沟通时你应该怎么表达

你可以直接用这套说法：

“如果我们底层设备平台已经稳定，实在Agent 可以作为上层智能运营与自动化平台，用来做查询、报表、告警处理、工单流转和跨系统编排。但它不应该承担设备接入、实时控制和核心联动引擎的职责。”

这句话很重要，因为它能直接把讨论拉回到正确架构。

## 六、你现在必须记住的判断原则

先记 4 句话：

1. 设备控制核心要靠 IoT 平台，不靠 RPA。
2. 实时联动要靠规则引擎，不靠桌面自动化。
3. 运营协同和跨系统串联，才是 Agent/RPA 的强项。
4. 先有稳定底座，再加智能入口，成功率最高。

## 七、这册学完后的自测题

你先不要急着看答案，自己写出来。

1. 一个智能家居管理平台最少可以粗分为哪几层？
2. 为什么“设备控制”和“运营自动化”不应该混在同一层处理？
3. 如果用户说“我要做一个能管理米家设备的平台”，你会先追问哪几个边界问题？
4. 为什么实在Agent 适合放在上层，而不是做控制核心？

## 八、练习任务

请你自己写 2 份输出：

### 练习 1：分层图

用文字列出你理解的 5 层结构，并在每层后面写一句话说明职责。

### 练习 2：边界表

把下面这些能力分成“底层 IoT 平台负责”或“上层 Agent/RPA 负责”两栏：

- 设备上线
- 查询离线设备
- 发送开灯指令
- 生成周报
- 烟感告警联动
- 自动创建工单
- 门锁状态同步
- 自然语言问答

## 九、这一册的结论

你这阶段最需要建立的不是工具熟练度，而是判断力。

只要边界判断对了，后面你再去学 API、RPA、实在Agent、PoC 设计，都会顺很多。

下一册应该学的是：设备模型与核心对象。也就是“设备、属性、指令、事件”到底是什么，以及为什么平台一定要围绕这些对象组织能力。
